## Universal Script Quality Practices (Applies to All Languages)
- Eliminate all commented-out "dead code" before production—use version control systems (Git, SVN) to track historical changes instead of leaving old code in comments
- Remove temporary debugging/logging code (print statements, console.log, debug flags) before production; replace with structured logging frameworks that can be enabled/disabled via configuration
- Implement structured error handlers appropriate for your language (try/catch, pcall/xpcall, set -e) to catch and manage exceptions gracefully
- Log meaningful error messages with enough context to diagnose issues (what failed, why it failed, what inputs were provided) without exposing sensitive data (passwords, API keys, internal paths)
- Validate all external inputs (command-line arguments, user input, file contents, API responses) for type, length, format, and range to prevent crashes and injection attacks
- Write self-explanatory code with descriptive naming conventions—only comment complex logic or non-obvious intent, not basic functionality
- Break scripts into small, single-purpose functions/modules (no longer than 20-30 lines each) to improve readability, testability, and reusability
- Replace duplicate code blocks with reusable functions or standard library calls to reduce maintenance burden
- Parameterize hard-coded values (file paths, API endpoints, thresholds) into command-line arguments, environment variables, or configuration files
- Run scripts with minimal required permissions (principle of least privilege); avoid root/admin access unless absolutely necessary
- Store sensitive credentials in secure managers (not hard-coded in scripts); never log sensitive data even in masked form
- Include header documentation (or README) explaining script purpose, dependencies, usage examples, and exit codes
- Version control all scripts with clear commit messages; tag stable releases for production use
- Write unit tests for critical logic covering edge cases, error conditions, and boundary values
- Use linters/formatters specific to your language to enforce consistent style automatically (e.g., black for Python, ESLint for JavaScript, shellcheck for Bash)
- Avoid deep nesting of conditional/loop statements (more than 3 levels deep) by extracting nested logic into separate functions or using early returns/guard clauses
- Integrate scripts with CI/CD pipelines: automate testing, linting, security scanning, and deployment on every commit/merge (use GitHub Actions, GitLab CI, Jenkins, etc.) to catch issues early and ensure consistent, repeatable deployments
- Implement centralized error monitoring and observability: use tools like Sentry, Datadog, New Relic, or Grafana to collect, aggregate, and alert on script errors in production (get real-time visibility into failures across all environments, reduce mean time to resolution (MTTR) for incidents)
- Write comprehensive CHANGELOG.md files following Keep a Changelog format: document all breaking changes, new features, bug fixes, and migration instructions for each version release (helps users track changes, plan upgrades, and troubleshoot issues)
- Use pre-commit hooks (via Husky, pre-commit framework, or equivalent) to automatically enforce best practices on every commit: run linters, formatters, security scanners, and unit tests before code is even pushed to the repository (catches issues early in the development cycle before they reach code review)
- Implement feature flags/toggles for risky changes: deploy new code behind flags that can be enabled/disabled without redeployment (reduces risk of production issues, enables canary deployments, A/B testing, and gradual rollouts)
- Document all environment variables and configuration options required by the script: include default values, allowed ranges, descriptions, and examples in a .env.example file or configuration documentation (eliminates confusion about setup requirements, reduces onboarding time for new developers)
- Use structured logging formats (JSON, Logfmt) instead of plain text logs: makes logs easier to parse, search, and analyze with log aggregation tools (ELK stack, Splunk, CloudWatch Logs) (improves debuggability, enables automated log analysis and alerting)
- Implement graceful shutdown handling: register signal handlers (SIGINT, SIGTERM) to allow scripts to finish current work, clean up resources, and exit cleanly when terminated (prevents data corruption, incomplete operations, or resource leaks from abrupt termination)
- Use lock files (package-lock.json, poetry.lock, Pipfile.lock, go.sum) to pin exact dependency versions across all environments: ensures deterministic builds, prevents "works on my machine" issues, and guarantees consistent behavior across development, staging, and production
- Perform regular dependency updates and security patching: schedule automated dependency update PRs using tools like Dependabot, Renovate, or Snyk to keep dependencies up-to-date and secure (reduces attack surface from vulnerable outdated libraries)
- Write runnable example scripts in documentation (not just code snippets): provide complete, working examples that users can copy and run immediately to understand how to use your script/library (improves user experience, reduces support requests)
- Use .editorconfig files to enforce consistent coding style (indentation, line endings, charset) across all editors and IDEs used by team members: eliminates "formatting wars", ensures consistent code appearance regardless of editor choice, and reduces code review noise around style issues
- Implement rate limiting and throttling for scripts that call external APIs: prevent hitting API rate limits, getting IP banned, or incurring unexpected costs from excessive API calls (protects against accidental or malicious overuse of third-party services)
- Design scripts with idempotent operations wherever possible: running the script multiple times with the same inputs should produce the same result without side effects (makes retries safe after transient failures, simplifies error recovery, and enables safe re-runs without duplicating data or causing inconsistencies)

## Lua (General)-Specific Script Best Practices
- Use local keyword for all variables and functions: global variable access is significantly slower than local access in Lua (Lua looks up locals in registers, while globals require hash table lookups), and locals avoid namespace pollution in larger projects (local variables are 10-100x faster to access than globals depending on use case)
- Use pcall (protected call) to wrap code that might throw errors: pcall returns a boolean status flag (true if no error, false if error occurred) followed by either the function return values or the error message (prevents unhandled errors from crashing the entire program, enables graceful error recovery)
- Use xpcall for more advanced error handling: xpcall takes two arguments: the function to call, and an error handler function that receives the error object and can process it (e.g., add stack trace information, log the error, convert it to a user-friendly message, or perform cleanup) (xpcall provides more flexible error handling than pcall, especially for library/embedding use cases)
- Remove commented-out code before production: use version control systems (Git, SVN) to track all changes to the codebase; commented-out code adds confusion, increases file size, and provides no functional value (commented-out code clutters the codebase, makes it harder to find active code, and can accidentally be re-enabled)
- Organize code into reusable modules using require() to load external Lua files: each module should return a table containing its public functions and variables (this follows Lua's module pattern, keeps code organized, enables code reuse across projects) (modules encapsulate functionality, reduce duplication, make large codebases manageable)
- Validate inputs early in functions (check for nil values, correct types, valid ranges) to fail fast with clear error messages instead of failing later with cryptic errors that are harder to diagnose (fail-fast principle: catch errors as close to the source as possible, provide actionable error messages)
- Use Lua tables for all data structures (arrays, dictionaries, namespaces, objects): tables are Lua's only composite data type, and mastering their use is key to writing idiomatic Lua code (tables are extremely flexible, can represent almost any data structure, and are highly optimized in Lua implementations)
- Keep functions short and focused on a single task (aim for functions no longer than 15-20 lines): short functions are easier to read, test, debug, and reuse across different parts of your application (single-responsibility principle: each function does one thing well)
- Avoid using global variables except for configuration values that must be accessed across multiple modules: global variables make code harder to test and reason about, and can cause unexpected side effects (global mutable state is a major source of bugs, makes testing difficult because tests can interfere with each other via globals)
- Use string.format() for building output strings instead of concatenation (.. operator): string.format is more readable for complex strings with multiple variables, and often performs better for large numbers of concatenations (string.format handles formatting, alignment, padding, and locale-specific conversions automatically)
- Use ipairs() for iterating over arrays (sequential integer keys starting at 1) and pairs() for iterating over dictionaries (non-integer or sparse keys): ipairs() is faster for array iteration because it uses internal array optimization (ipairs uses numeric for loop internally, pairs uses generic hash table iteration which is slower for arrays)
- Avoid using the # operator to get the length of tables with holes (missing integer keys): # only counts contiguous integer keys starting at 1, so it may return incorrect results for sparse arrays; use a custom length function if you need to count all elements (# operator is optimized for dense arrays, but gives wrong results for sparse tables)
- Use metatables (__index, __newindex, __tostring, __len__, etc.) to implement object-oriented programming patterns, operator overloading, or read-only tables: metatables are one of Lua's most powerful features for extending language behavior (metatables enable OOP, custom data structures, domain-specific languages, and many other advanced patterns)
- Close file handles and other resources explicitly using io.close() or in a finally-like pattern (using pcall/xpcall to ensure cleanup runs even if errors occur): Lua does not have garbage collection guarantees for immediate resource cleanup, so explicit closing is important for resources like files and network connections (resource leaks can exhaust file descriptors, network sockets, or memory over time)
- Use coroutine.yield() and coroutine.resume() for cooperative multitasking: coroutines allow you to pause and resume function execution, which is useful for implementing iterators, state machines, concurrent tasks without threads (coroutines are lightweight, cooperative multitasking: they yield control voluntarily, avoiding threading complexity)
- Avoid using loadstring() or load() with untrusted input: these functions execute arbitrary Lua code, which poses a major security risk if the input comes from an untrusted source (user input, network requests, file contents) (loadstring/load can execute arbitrary code, enabling remote code execution attacks)
- Use math.floor() / math.ceil() / math.round() for explicit numeric rounding instead of relying on implicit conversion: Lua's number type is floating-point double precision by default, so be aware of precision issues when working with decimal numbers (floating-point arithmetic can produce unexpected rounding errors, explicit rounding makes intent clear)
- Comment code only to explain non-obvious intent, algorithm choices, workarounds for platform limitations, or complex business rules: do not comment obvious code (e.g., i = i + 1 -- increment i) as this adds noise and reduces readability (comments should explain *why* not *what*: code should be self-explanatory for *what* it does)
- Use os.time() and os.date() for date/time operations instead of trying to calculate time manually: these functions handle timezone conversions, leap years, and other date/time complexities correctly (manual date calculations are error-prone, fail to account for timezone rules, leap seconds, etc.)
- Test your Lua scripts in multiple Lua implementations (Lua 5.1, 5.3, 5.4, Luau, LuaJIT) if you target multiple platforms: there are small differences between implementations that can cause compatibility issues (different implementations have slightly different standard libraries, performance characteristics, and language features)
- Use the debug library (debug.traceback, debug.getinfo) sparingly and only in development/debugging code: debug functions have significant performance overhead and should not be included in production code (debug functions can slow down code by orders of magnitude, expose internal implementation details)
- Avoid modifying tables while iterating over them (adding/removing elements during ipairs/pairs loop): this can cause skipped elements or infinite loops; instead, collect the elements to modify in a separate table, then modify the original table after the loop completes (modifying tables during iteration leads to undefined behavior, missed elements, or infinite loops)
- When embedding Lua in C/C++ applications, use lua_pcall/lua_xpcall for all Lua code execution from C to catch and handle errors gracefully: prevents unhandled Lua errors from crashing the host application (Lua errors propagate to C as longjmp by default, which crashes the host if not caught; pcall/xpcall convert errors to return values that C code can handle normally)
- Use lua_newuserdata to store C data structures in Lua objects with proper metatables for garbage collection: prevents memory leaks when Lua objects holding C pointers are collected (userdata allows C data to be managed by Lua's GC, metatable __gc callback frees C memory when userdata is collected)
- Pre-allocate Lua tables with known sizes using lua_createtable(L, narr, nrec) when creating large tables from C: significantly improves table creation performance compared to incremental insertion (pre-allocation avoids repeated resizing and copying as table grows, which is O(n²) for incremental insertion vs O(n) for pre-allocated)
- Use lua_rawget/lua_rawset instead of lua_gettable/lua_settable when you don't need metamethod invocation: raw access is faster as it skips metatable lookups (raw operations skip the metatable chain, providing significant performance gains for hot paths in C/Lua binding code)
- Avoid using lua_next on tables while modifying them (can cause skipped entries or crashes); instead, collect keys into an array first, then iterate over the array (lua_next assumes table structure doesn't change during iteration; modifying the table during iteration leads to undefined behavior, crashes, or missed entries)
- Use lua_load with a custom reader function to load Lua scripts from memory buffers or custom sources (instead of only loading from files): enables embedding Lua scripts in C application resources, databases, or network streams (custom readers allow loading scripts from any source, not just filesystem files, essential for embedded applications)
